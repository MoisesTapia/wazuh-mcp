# SOC & SOAR

Beyond the Manager REST API, the server ships a **SOC layer** (alerts and
vulnerabilities queried directly from the Wazuh Indexer) and a **SOAR layer**
(active response actions).

## Why the Indexer?

Since **Wazuh 4.8.0**, alerts and vulnerability state are stored **only** in the
Wazuh Indexer (OpenSearch, port 9200). The Manager REST API (port 55000) no
longer exposes them. The SOC tools therefore talk to the Indexer directly using
HTTP Basic Auth against these indices:

| Index pattern | Contents |
|---------------|----------|
| `wazuh-alerts-4.x-*` | Alerts generated by the manager |
| `wazuh-states-vulnerabilities-*` | CVEs detected per agent |
| `wazuh-archives-4.x-*` | All events (unfiltered) |
| `wazuh-monitoring-*` | Historical agent state |

The Indexer client (`src/wazuh_mcp/api/wazuh_indexer.py`) exposes Elasticsearch
DSL query helpers (`time_range_query`, `term_query`, `bool_query`, …) and
redacts secrets from alert documents before they reach the LLM.

## Enabling the SOC tools

Set `WAZUH_INDEXER_HOST` (and related variables) in `.env`:

```env
WAZUH_INDEXER_HOST=localhost
WAZUH_INDEXER_PORT=9200
WAZUH_INDEXER_USER=admin
WAZUH_INDEXER_PASSWORD=SecretPassword
WAZUH_INDEXER_VERIFY_SSL=false
```

See [Configuration](configuration.md#wazuh-indexer-soc-tools). When running the
MCP inside Docker (`make mcp-docker`), the Indexer connection is pre-wired in
`docker-compose.yml`.

## SOC — alerts

Module `soc_alerts` (7 tools):

| Tool | Purpose |
|------|---------|
| `get_alerts` | Recent alerts, filterable by time window and rule level |
| `get_critical_alerts` | High-severity alerts only |
| `get_alert_summary` | Aggregated counts by rule, agent, severity |
| `search_alerts` | Full-text / field search across alerts |
| `get_agent_alert_timeline` | Chronological alerts for a single agent |
| `get_top_threats` | Ranked threats across the fleet |
| `analyze_alert_patterns` | Pattern / trend analysis over a period |

```
"Show me critical alerts from the last 6 hours"     → get_critical_alerts(hours=6)
"What are today's top threats?"                      → get_top_threats()
"Timeline of alerts for agent 001"                   → get_agent_alert_timeline(agent_id="001")
```

## SOC — vulnerabilities

Module `soc_vulnerabilities` (5 tools):

| Tool | Purpose |
|------|---------|
| `get_vulnerabilities` | CVEs, filterable by agent and severity |
| `get_critical_vulnerabilities` | Critical CVEs only |
| `get_vulnerability_summary` | Aggregated CVE counts |
| `get_agent_risk_score` | Composite risk score for one agent |
| `search_cve` | Look up a specific CVE across the environment |

```
"Which agents are affected by CVE-2024-3094?"  → search_cve(cve_id="CVE-2024-3094")
"Risk score for agent 001"                      → get_agent_risk_score(agent_id="001")
```

## SOAR — active response

Module `active_response_soc` (14 tools) turns Wazuh Active Response into
guided SOAR actions, plus verification helpers to confirm each action landed.

**Actions:**

| Tool | Purpose |
|------|---------|
| `wazuh_block_ip` | Block an IP on an agent (firewall-drop) |
| `wazuh_unblock_ip` | Remove a previously blocked IP |
| `isolate_agent` | Network-isolate an agent |
| `unisolate_agent` | Restore agent network access |
| `kill_process` | Terminate a process on an agent |
| `restart_wazuh_agent_ar` | Restart the Wazuh agent via AR |
| `run_yara_scan` | Trigger a YARA scan on an agent |
| `run_custom_ar` | Run a custom AR script |
| `wazuh_add_ip_to_cdb` | Add an IP to a CDB list (e.g. blocklist) |

**Verification helpers:**

| Tool | Purpose |
|------|---------|
| `check_blocked_ip` | Confirm an IP is actually blocked |
| `check_agent_connectivity` | Confirm agent reachability after an action |
| `get_agent_active_responses` | List AR executions on an agent |
| `verify_process_killed` | Confirm a process is no longer running |
| `get_ar_scripts_available` | List AR scripts installed on the agent |

> **Destructive by design.** These tools change the state of monitored hosts.
> Their docstrings are tagged `DESTRUCTIVE:`/`CAUTION:`, so the audit log flags
> them and Claude is instructed to confirm with the user first. See
> [Security model](security.md#audit-logging).

```
"Block 192.168.1.100 on agent 001"   → wazuh_block_ip(agent_id="001", ip="192.168.1.100")
"...then confirm it worked"           → check_blocked_ip(agent_id="001", ip="192.168.1.100")
```

## Graceful degradation

If `WAZUH_INDEXER_HOST` is not configured, the SOC tools do **not** crash. They
return a structured "not configured" response:

```json
{
  "error": "Wazuh Indexer no configurado",
  "not_configured": true,
  "help": "Añade WAZUH_INDEXER_HOST al .env para habilitar esta tool."
}
```

`get_mcp_health` reports the Indexer as `not_configured` in that case, and as
`degraded` if it is configured but unreachable. See
[Architecture](architecture.md) and [Observability](security.md).
